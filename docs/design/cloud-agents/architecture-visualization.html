Cloud Agents Framework

G1
Bring Your Own Agents & Workflows
YAML + any tools. No forking, no rebuilds, no framework changes. Product teams deploy AI agents without changing framework code.
G2
Secured & Governed Execution
Ephemeral containers, scoped permissions, human approval gates. Full observability: tracing, metrics, event streaming.
G3
Composable Agent Ecosystem
Agents and workflows are reusable building blocks. Multiple trigger points: conversations, alerts, API, schedules.
G4
Seamless Human-Agent Handoff
When automation reaches its limit, users continue in an AI CLI with full workflow context.
G5
Dual Deployment: Kubernetes + Podman
Same agents run on both. Both are production targets.
Design Principles
R1 Framework, not pre-built agents G1
Examples ship as examples. Framework provides runtime, executor, spawner, and observability. Teams provide agent.yaml, tools, and workflow definitions. One generic container image for all agent types.
R2 Multi-step workflows with human oversight G1 G2
Chain agents with conditions, retry, approval gates, escalation. Low-risk steps auto-approve; high-risk steps require human review.
R3 Ephemeral-by-default G2
Fresh container per step. Only configured tools loaded. Hard timeouts. Destroyed after execution. Untrusted pods never receive secrets beyond their API token. Pre-deployed option for latency-sensitive steps.
R4 Human-in-the-loop G2
Diagnose → Propose → Gate → Execute → Verify. Structured output: confidence, risk, rollback plan. Approval gates pause workflows for human review.
R5 Human-out-of-the-loop G2 G4
When retries exhaust, escalation packages full context — diagnosis, steps taken, failure history, tools, cluster state. User picks up in an interactive AI CLI session and continues investigating where the agent left off.
R6 Retry with escalation G2 G4
Each retry sees full failure history. Exhausted → R4 (approval) or R5 (interactive handoff).
R7 Stateless runner, durable state G1 G5
No in-memory state. Durable external store. Scales horizontally behind a load balancer. Replica crashes don't lose workflows.
Technical Requirements
R8 Execution engine G1 G2
Temporal workflow engine for durable orchestration. lightspeed-agentic-sandbox for ephemeral agent containers with OpenAI agents SDK.
R9 Runtime G1
OpenAI agents SDK (in sandbox) + FastAPI + Temporal workflow engine
R10 Deployment G5
K8s: Jobs, Services, ConfigMaps, Secrets, RBAC
Podman: containers, volume mounts, port mapping
R11 Persistence G1
Durable store with optimistic locking. Atomic reads/writes, pluggable backend.
R12 Security G2
Secrets via K8s refs, bearer auth on all endpoints. Explicit risk_level, per-step permission scoping.
R13 Access control G2
RBAC for workflows: who can trigger, approve, view. RBAC for tools: which tools each agent/user can invoke. Scoped by team, role, or namespace.
R14 Observability G2
OTel tracing, Prometheus metrics, SSE streaming
R15 Triggers G3
Chatbot, alerts, API, scheduled events
R16 Agents-as-tools G3
Registry auto-generates tools from definitions
R17 Escalation G4
Context packaging: workflow state, failure history, tools, cluster context → CLI handoff
Frequently Asked Questions
What is the schema of the workflow definition (workflow.yaml)? Is it standardized?
Custom YAML schema (apiVersion: v1, kind: AgentWorkflow). Not an industry standard — there isn't one for agentic workflows yet. The schema defines steps with type (agent/human-approval), conditions, retry, parallel groups, risk level, and spawn mode. It's a Pydantic model so validation is strict and errors are clear. We should evaluate alignment with emerging standards (e.g. A2A protocol) as they mature.
How are agents going to be defined? Custom YAML schema?
Custom YAML (apiVersion: lightspeed.redhat.com/v1alpha1, kind: AgentDefinition). Defines: system prompt, tools to load, output type, lifecycle (request-response or periodic-loop), resource limits. The schema is intentionally minimal — just enough to configure the generic runtime. No code generation, no framework lock-in.
Where are agent guardrails defined? What guardrails will be available?
Multiple layers:
• Workflow level: risk_level per step, approval gates, advisory mode (read-only tool filtering)
• Agent level: allowed_tools / denied_tools per step via PermissionScope
• Runtime level: hard timeouts, resource limits (CPU/memory), ephemeral container isolation
• Cluster level: K8s RBAC, NetworkPolicy, ServiceAccount scoping
• Future: RBAC for who can trigger/approve/view workflows (R12)
Are these agents going to be available as a ready-made template?
Example agents ship as templates in examples/agents/: diagnostic, monitoring, deploy-readiness. Teams copy and modify. The framework itself is not a marketplace — it's a runtime. Template distribution (OCI artifacts, git-sync) is a backlog item.
Are teams able to bring their own agent? Do they need a specific SDK or just a standardized API?
Both paths:
• YAML + tools: Define agent via YAML, bring Python/bash/CLI tools. Framework runs them in the generic runtime. No SDK needed.
• Standardized API: Any service exposing a POST /v1/agent/run HTTP endpoint (query + context in, structured output out) can be called as an agent. No framework dependency. This aligns with the direction of A2A/OpenAI-compatible APIs.

Does it support durable agents with long-running tasks?
Yes, two modes:
• Pre-deployed agents (spawn: pre-deployed): long-running services that persist across workflow executions. Called via HTTP, not spawned/destroyed per step. Suitable for agents that maintain state, connections, or caches.
• Temporal durable execution (proposed): workflow state survives crashes, restarts, and arbitrarily long waits (e.g. approval gates that take days). The agent step itself runs in an ephemeral container, but the workflow orchestration is durable.

What's missing: agents that maintain conversational memory across multiple workflow invocations. That's a backlog item (agent state persistence / memory).
